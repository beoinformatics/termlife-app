# TermLife Command Language & History Architecture

## Overview

TermLife defines an internal command language (TCL — TermLife Command Language) that provides a unified, text-based interface for every action the terminal app can perform. This language serves as the foundation for tab history, command replay, and the command palette.

---

## Design Philosophy

Every UI interaction in TermLife — whether a mouse click, keyboard shortcut, or typed command — is reducible to a single canonical text representation. This creates a **single source of truth** for what happened in a tab, enabling:

1. **History** — a complete, reviewable log of user actions per tab
2. **Replay** — deterministic recreation of a session from a saved file
3. **Command Palette** — VS Code-style searchable command entry
4. **Scriptability** — batch execution via replay files

---

## Two Types of History Entries

### Shell Commands (plain text)

Lines the user typed at the shell prompt. Captured only when the tab's PTY has **no child processes** (i.e., the shell is idle and awaiting input). This ensures:

- Commands like `cd foo`, `git status`, `npm install` are captured
- Keystrokes inside interactive programs (vim, python REPL, less) are NOT captured
- This is a deliberate, conservative tradeoff: reliability over completeness

### App Commands (bracket syntax)

Actions directed at TermLife itself, enclosed in square brackets:

```
[category:action arguments]
```

These are generated by:
- Mouse clicks on UI elements (buttons, tabs, dashboard)
- Keyboard shortcuts
- Command palette input

---

## TermLife Command Language (TCL)

### Syntax

```
[category:action arg1 arg2 ...]
```

- **category** — the subsystem (tab, split, theme, crt, matrix, dashboard, history, scroll)
- **action** — the operation within that subsystem
- **arguments** — space-separated, optional depending on the command

### Command Registry

| Command | Description |
|---------|-------------|
| `[tab:new]` | Create new tab (single layout) |
| `[tab:new vertical]` | Create new tab with vertical split |
| `[tab:new horizontal]` | Create new tab with horizontal split |
| `[tab:new quad]` | Create new tab with quad split |
| `[tab:close]` | Close current tab |
| `[tab:switch N]` | Switch to tab N (1-indexed) |
| `[tab:next]` | Switch to next tab (cycles) |
| `[tab:prev]` | Switch to previous tab (cycles) |
| `[tab:smart-switch]` | Smart tab switching (Ctrl+Tab logic) |
| `[split:single]` | Change to single pane layout |
| `[split:vertical]` | Change to vertical split |
| `[split:horizontal]` | Change to horizontal split |
| `[split:quad]` | Change to quad layout |
| `[theme:set NAME]` | Set color theme by name |
| `[theme:list]` | Print available theme names to terminal |
| `[theme:cycle]` | Cycle to next theme |
| `[crt:toggle]` | Toggle CRT filter |
| `[crt:on]` | Enable CRT filter |
| `[crt:off]` | Disable CRT filter |
| `[matrix:toggle]` | Toggle Matrix rain effect |
| `[matrix:on]` | Enable Matrix rain |
| `[matrix:off]` | Disable Matrix rain |
| `[dashboard:toggle]` | Toggle dashboard view |
| `[dashboard:open]` | Open dashboard |
| `[dashboard:close]` | Close dashboard |
| `[history:toggle]` | Toggle history view for current tab |
| `[history:open]` | Open history view |
| `[history:close]` | Close history view |
| `[scroll:top]` | Jump to top of scrollback |
| `[scroll:bottom]` | Jump to bottom of scrollback |
| `[scroll:up N]` | Scroll up N lines |
| `[scroll:down N]` | Scroll down N lines |

The registry is extensible — new features register their commands here.

### Parsing Rules

1. Lines starting with `[` and ending with `]` are app commands
2. All other lines are shell commands
3. Empty lines are ignored
4. Lines starting with `#` are comments (in replay files only)

## History System

### Per-Tab History

Each tab maintains its own `TabHistory` instance. History is:

- **Ephemeral** — lives only in memory, not persisted to disk automatically
- **Tab-isolated** — no sharing between tabs (security: tabs may contain sensitive data)
- **Append-only** — entries are timestamped and ordered chronologically

### History Entry Structure

```typescript
interface HistoryEntry {
  timestamp: number          // Date.now()
  type: 'shell' | 'app'     // Distinguishes the two kinds
  command: string            // The text (plain for shell, bracketed for app)
}
```

### Shell Command Capture

Shell commands are captured by observing:

1. The user presses Enter (detected in InputHandler)
2. The tab's PTY currently has **no child processes** (hasChildren === false)
3. The accumulated input buffer since the last Enter is recorded as a shell command
4. The input buffer is reset

This means:
- Interactive programs (vim, node REPL, python) are NOT recorded — their keystrokes happen while hasChildren is true
- Only top-level shell commands at the prompt are captured
- Multi-line commands (heredocs, backslash continuations) may be captured as multiple entries — acceptable tradeoff

### App Command Capture

Every app command execution logs itself to the active tab's history. This happens at the command dispatch layer, ensuring all entry points (mouse, keyboard, palette) produce the same history entry.

## Command Mode (Inline Palette)

### Activation

`Cmd/Ctrl+Shift+P` enters command mode. The terminal prompt changes from `>` to `:`, indicating the next input line is an app command rather than a shell command.

### Behavior

1. User presses `Cmd/Ctrl+Shift+P` — a `:` prompt appears on a new line
2. Keystrokes are collected locally and echoed to the terminal (not sent to PTY)
3. `Enter` executes the command and exits command mode
4. `Escape` cancels and exits command mode
5. `Backspace` and `Ctrl+U` work for editing
6. Commands are matched first as exact TCL IDs (e.g. `theme:set dracula`), then by fuzzy search
7. Executed command is logged to the tab's history

### Three Entry Points

Every app action is accessible via exactly three paths:

| Entry Point | Example |
|-------------|---------|
| **Mouse** | Click the "⧧" button in the tab bar |
| **Keyboard shortcut** | Press the bound key combination |
| **Command mode** | Press `Cmd/Ctrl+Shift+P`, then type `split:vertical` at the `:` prompt |

## Replay System

### Saving a Session

1. User opens the history view for a tab
2. Selects and copies history entries (or all)
3. Saves to a text file

### Replay File Format

Plain text, one entry per line:

```
# TermLife session replay
# Saved: 2026-03-06 14:30:00

cd ~/projects/termlife-app
npm install
[split:vertical]
npm run dev
[theme:set dracula]
git status
[crt:toggle]
```

### Replay Execution

Invoked via CLI flag:

```bash
termlife -f replay.txt
termlife --replay replay.txt
```

Replay logic:

1. Read file line by line
2. Skip empty lines and `#` comments
3. For lines starting with `[` → parse and execute as app command
4. For all other lines → send to the shell as typed input + Enter
5. **Wait for idle** — after sending a shell command, wait until hasChildren transitions back to false before sending the next line
6. App commands execute immediately (they don't involve the shell)

### Limitations

- Interactive programs cannot be replayed (by design — they were never captured)
- Commands that prompt for input (e.g., `rm -i`) will block replay
- Environment-dependent commands may behave differently on replay
- No error handling during replay — if a command fails, replay continues

## History View

### Activation

- Keyboard: bound shortcut or `[history:toggle]` via palette
- Shows a scrollable panel replacing or overlaying the terminal content

### Rendering

```
 14:30:01  cd ~/projects/termlife-app
 14:30:03  npm install
 14:30:15  [split:vertical]
 14:31:22  git status
 14:31:30  [theme:set dracula]
 14:32:01  [crt:toggle]
```

- **Timestamps** on the left in a muted color
- **Shell commands** in the terminal's normal text color
- **App commands** in an accent/dimmed color, italicized
- Text is selectable for copy-paste
- Escape or `[history:close]` dismisses the view

---

## Security Considerations

- Tab histories are never shared across tabs
- No automatic persistence — sensitive commands don't leak to disk
- Replay files are user-created and user-managed
- The command palette cannot execute arbitrary shell commands (only registered app commands)
